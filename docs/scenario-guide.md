# Scenario Guide

Use this guide to choose the smallest safe command for common engineering requests.

| Scenario | Use When | Start Command | Key Evidence | Next Step |
| --- | --- | --- | --- | --- |
| `one_line_request` One-line request | A small user request arrives as one sentence or a short chat message. | `python3 scripts/codex_eng.py auto --input requirement.md --out /tmp/codex-auto` | `auto_run_summary.json`, `spec.json`, `delivery_plan_review.json` | Read auto_run_summary.json for the selected profile, blockers, and next safe command. |
| `long_prd` Long PRD | A copied PRD, Markdown document, or long requirement needs normalization before design. | `python3 scripts/codex_eng.py auto --input prd.md --out /tmp/codex-auto` | `spec.json`, `technical_design.json`, `architecture_design.json`, `delivery_plan.json` | Resolve open questions first; otherwise continue with the next command in auto_run_summary.json. |
| `bugfix` Bugfix | A defect or regression needs a lightweight but gated fix path. | `python3 scripts/codex_eng.py auto --input bug.md --profile bugfix --out /tmp/codex-auto` | `spec.json`, `technical_design.json`, `delivery_plan_review.json` | Add reproduction, Git readiness, edit permit, implementation, review, and test evidence before release. |
| `frontend_change` Frontend change | A page, route, form, table, export, or browser-visible behavior changes. | `python3 scripts/codex_eng.py auto --input ui.md --profile frontend_change --out /tmp/codex-auto` | `frontend_acceptance.json`, `test_evidence_gate.json`, `auto_run_summary.json` | Replace template browser evidence with real frontend acceptance before review or release. |
| `cross_repo_api` API or cross-repo change | A backend API, route contract, producer/consumer boundary, or existing repository context matters. | `python3 scripts/codex_eng.py auto --input api.md --repo /path/to/repo --project project-name --out /tmp/codex-auto` | `project_understanding/baseline_quality.json`, `technical_design.json`, `delivery_plan_review.json` | Use project understanding and traceability evidence before implementation. |
| `data_migration` Data migration | A database, migration, sensitive data, configuration, performance, or rollback-sensitive change is required. | `python3 scripts/codex_eng.py auto --input data.md --profile data_migration --out /tmp/codex-auto` | `configuration_readiness.json`, `data_security_review.json`, `performance_review.json`, `release_gate.json` | Complete real security, performance, configuration, rollback, and release evidence before implementation/release. |
| `release_readiness` Release readiness | Implementation exists and the question is whether release is allowed. | `python3 scripts/codex_eng.py auto --input release.md --profile release_readiness --out /tmp/codex-auto` | `implementation_completion_gate.json`, `code_review_gate.json`, `test_evidence_gate.json`, `release_gate.json` | Fill missing release artifacts until release_gate.json returns go or conditional_go. |
| `code_review` Code review | A diff exists and needs design-quality, risk, test, and release evidence review. | `git diff > /tmp/codex.diff && python3 scripts/codex_eng.py run diff-impact --diff-file /tmp/codex.diff --out /tmp/codex-diff-impact.json` | `diff_impact.json`, `code_review_gate.json`, `test_evidence_gate.json` | Bind diff impact, write guard, code review, test, CI, and release evidence before approval. |

For normal requirement handling, prefer `python3 scripts/codex_eng.py auto --input requirement.md` first.
Use explicit `--profile` only when the scenario is already known or the automatic profile choice needs to be constrained.
